# 客服回复幻觉检测工具

## 1. 幻觉分类体系

### 1.1 分类总览

| 类型 | 严重程度 | 定义 | 示例 |
|---|---|---|---|
| **安全误导** | 🔴 Critical | 忽视风险提示，给出可能危害用户健康/安全的建议 | 含视黄醇的面膜却说孕妇可放心使用 |
| **参数编造** | 🟠 High | 编造产品核心参数（材质、规格、功能） | PU 合成革说成头层牛皮；蓝牙 5.0 说成 5.3 |
| **政策编造** | 🟠 High | 虚构更宽松的交易政策 | 7 天无理由说成 30 天无理由且包运费 |
| **优惠编造** | 🟡 Medium | 虚构不存在的优惠活动 | 编造满 300 减 50 券、不存在的学生优惠 |
| **能力越界** | 🟡 Medium | 假装具备系统不具备的能力并给出具体信息 | 未接入物流接口却声称查到包裹在南京 |
| **信息编造** | 🟡 Medium | 编造知识库中不存在的事实信息 | 纯线上品牌说有线下门店；给虚假退货地址 |
| **政策偏差** | 🔵 Low | 部分正确但存在关键细节错误 | 发货时间从 24h 说成 48h；快递公司张冠李戴 |
| **信息遗漏** | 🔵 Low | 忽略知识库中的关键信息导致建议不准确 | 30% 用户反馈偏大半码，却说尺码标准 |

### 1.2 严重程度定义

```
Critical (安全误导)  → 可能造成人身伤害，需零容忍
High    (参数/政策编造) → 核心商业信息错误，可能导致交易纠纷
Medium  (优惠编造/能力越界/信息编造) → 影响用户体验和信任
Low     (政策偏差/信息遗漏) → 部分错误或信息不完整
```

### 2.2 为什么是「规则 + LLM」混合方案？

检测客服幻觉本质上是一个**高精度、低成本、可解释**的工程问题。评估了 4 种候选方案：

| 方案 | 准确率 | 单条成本 | 延迟 | 可解释性 | 冷启动 |
|---|---|---|---|---|---|
| **A. 纯规则** | ⭐⭐ 中等 | 免费 | < 1ms | ✅ 每条都有明确理由 | 需大量人工编写 |
| **B. 纯 LLM** | ⭐⭐⭐ 较高 | ~¥0.01/条 | 2~6s | ❌ 黑盒判断 | 零（调 API 即用） |
| **C. Embedding 语义相似度** | ⭐⭐ 中等 | 低 | ~100ms | ❌ 只有相似度分数 | 需标注数据训练阈值 |
| **D. 混合：规则 + LLM** ✅ | ⭐⭐⭐ 高 | 极低 | < 1ms~6s | ✅ 规则可解释 | 规则覆盖常见模式 |

**逐一排除：**

- **纯规则（A）**：客服对话千变万化，"我查了下您的包裹，现在到南京了"和"帮您看了一下，快递在南京中转"意思相同但措辞完全不同。纯正则无法穷举所有变体，**漏报率会随数据多样性急剧上升**。

- **纯 LLM（B）**：20 条 ≈ ¥0.2，但 20 万条 ≈ ¥2000。而且每条 2~6 秒的延迟对实时客服场景不可接受。更重要的是，LLM 对「知识库明确说 A、回复说 B」这类**确定性矛盾**反而显得"大材小用"——就像用火箭炮打蚊子，浪费算力。

- **Embedding（C）**：适合发现「回复和 KB 语义不相似」，但这类幻觉中很多是「回复和 KB 高度相似，只有关键数字不同」（如 7 天→30 天，48h→24h）。Embedding 对数字不敏感，**精确的数值差异会被语义相似度淹没**。

- **混合方案（D）**：规则负责「确定性矛盾」（KB 说 7 天，回复说 30 天——这种没有歧义），LLM 负责「模糊边界」（需要理解上下文、判断是否有隐含冲突）。**规则拦截了 90% 的常规 case，LLM 只需处理剩下的 10%**。

**核心设计原则：**

```
确定性矛盾 → 规则（快、免费、精确）
模糊边界   → LLM（理解上下文、处理变体）

不是"规则不行了才用 LLM"，而是"规则能确定的就不浪费 LLM"
```

### 2.3 规则引擎（主力，命中率 90%）

**原理**：针对 8 类幻觉，逐一编写基于正则和关键词的检测规则。

每条规则遵循「**KB 断言 + Reply 矛盾 → 触发**」的二段式逻辑：

| 幻觉类型 | 规则逻辑 | 代表 case |
|---|---|---|
| 政策编造 | KB 和 Reply 对退货天数描述冲突 | h01: 7天 vs 30天 |
| 参数编造 | 蓝牙版本/接口类型/材质/保修期/NFC 与 KB 不一致 | h02: 蓝牙 5.0 vs 5.3 |
| 能力越界 | KB 声明"未接入/不具备" + Reply 有伪装执行信号 | h03: 未接入物流却报位置 |
| 政策偏差 | 发货时间/快递公司/发票类型/申请方式与 KB 冲突 | h04: 纸质发票 vs 暂不支持 |
| 优惠编造 | Reply 中的优惠在 KB 中被明确否定或列出的清单中不存在 | h05: 满300减50 vs 无此活动 |
| 信息编造 | Reply 声称存在 KB 明确说没有的实体信息（门店/地址/品牌关系） | h11: 有线店 vs 纯线上 |
| 安全误导 | Reply 面向敏感人群称"放心" + KB 含风险提示 | h13: 孕妇放心 vs 含视黄醇 |
| 信息遗漏 | KB 有偏差信息 + Reply 称"标准/不偏" | h20: 30%偏大 vs 尺码标准 |

**v2.1 改进**：支持**多类型检测**——同一条回复可能命中多条规则，不再提前 return。当前测试集检出 h04、h08 各含 2 个问题。

### 2.4 LLM 检测（兜底，命中率 10%）

**原理**：规则未命中的 case 交由 LLM 判断。LLM 以「文本事实核查工具」的角色对比回复和知识库，输出结构化 JSON。

**配置**：
- API 端点：`https://api.agicto.cn/v1`
- 模型：`gpt-5-chat-latest`
- Temperature：0.1（低随机性，适合事实核查）
- Mock 模式：`config.MOCK_MODE = True` 可跳过 API 调用跑通流程（开发调试用）

**容错策略**：

| 异常情况 | 降级策略 |
|---|---|
| API 返回空内容（安全拦截） | 按幻觉处理（宁漏报勿误报） |
| 响应非 JSON 格式 | 正则提取 `{...}` 后再解析 |
| JSON 解析失败 | 按幻觉处理 + 记录原始内容 |
| 网络异常 | 返回 None → detector 标记为 fallback |

### 2.5 评估方法

`evaluate.py` 对比 `result.json`（预测）和 `task4_ground_truth.json`（人工标注），计算：

- **混淆矩阵**：TP / FP / FN / TN
- **准确率** (Accuracy)：`(TP+TN) / Total`
- **精确率** (Precision)：`TP / (TP+FP)`
- **召回率** (Recall)：`TP / (TP+FN)`
- **F1 值**：精确率和召回率的调和平均

## 4. 误判分析

> 当前测试集 20 条实现了零误判零漏检。以下分析基于规则设计边界，预测**哪些类型的 case 在未来最容易出现误判**。

### 4.1 易误报场景（正常回复被判为幻觉）

| 场景 | 风险点 | 对应规则 | 缓解措施 |
|---|---|---|---|
| **诚实拒绝含数字** | 客服诚实说"我无法查询，请拨 400 电话"，回复中的电话号码可能触发误判 | 能力越界 | v2.1 已加入诚实拒绝排除层（关键词：`无法/不能/暂无此功能`） |
| **知识库不完整** | KB 说"未标注 NFC"，但产品实际有 NFC（KB 本身过时），规则会误判为参数编造 | 参数编造 | 需 KB 维护流程配合，或对"未标注"降低置信度 |
| **快递公司别名** | KB 说"中通/韵达/圆通"，回复说"申通"——不在关键词列表中的快递公司不会被规则捕获差异 | 政策偏差 | 快递公司关键词列表需要持续维护 |
| **合理的政策变化** | 如果退货政策真的从 7 天改成了 30 天，规则仍会判为幻觉 | 政策编造 | 需配合 KB 更新机制或加入时效性检查 |

### 4.2 易漏报场景（幻觉未被检出）

| 场景 | 风险点 | 涉及规则 | 缓解措施 |
|---|---|---|---|
| **话术层面的误导** | "这个商品很受欢迎呢~"——模糊的正面评价，不能算事实错误但具误导性 | 无对应规则 | 需单独建立"营销话术夸大"类别 |
| **隐性能力越界** | 回复说"我帮您记录反馈了"——如果 KB 没声明无此能力，规则不会触发 | 能力越界 | 依赖 KB 对系统能力的声明完整性 |
| **细微数字偏差** | 回复说"约 3-4 天到货"，KB 说"3-5 天"——范围重叠但用词不同 | 政策偏差 | 当前规则不做数值范围比较，需增强 |
| **上下文省略** | 回复只说结论不提限制条件（如"可以退货"但省略了"需未拆封"） | 信息遗漏 | 当前只有尺码偏差检测，覆盖面窄 |

### 4.3 最难判断的边界 Case

**Top 3 棘手场景**：

1. **h13（安全误导）类型**：需要专业知识判断（如视黄醇对孕妇的风险）。规则依赖 KB 中恰好有"慎用/咨询医生"等关键词，如果 KB 用词更委婉，规则就会漏。

2. **h20（信息遗漏）类型**：KB 有数据（30% 用户反馈偏大），回复简化后丢失了关键 nuance。这类"省略式幻觉"需要理解 KB 中统计信息的含义，纯规则难以泛化。

3. **h04（半真半假）类型**：回复部分正确（电子发票 ✓）部分错误（纸质发票 ✗），规则需要拆解到子命题级别才能准确判断，当前实现将两个问题都标记为"政策偏差"而未区分正确/错误部分。

## 5. 项目结构

## 6. 快速开始

### 6.1 环境要求

- Python 3.10+
- `openai` 包（仅真实 API 模式需要）

```bash
pip install openai
```

### 6.2 运行检测

```bash
# 进入项目目录
cd 客服agent试题

# Mock 模式（不消耗 API，快速验证流程）
python -c "import config; config.MOCK_MODE = True; import detector; detector.main()"

# 真实 API 模式（需先配置 config.py 中的 API_KEY）
python detector.py
```

### 6.3 运行评估

```bash
# 评估检测结果 vs 人工标注
python evaluate.py
```

### 6.4 配置说明

编辑 `config.py`：

```python
# Mock 模式开关
MOCK_MODE = False          # True = 跳过 API，False = 真实调用

# API 配置
BASE_URL = "https://api.agicto.cn/v1"
API_KEY = "your-api-key"   # 建议用环境变量: export OPENAI_API_KEY=xxx
MODEL_NAME = "gpt-5-chat-latest"

# 模型参数
TEMPERATURE = 0.1          # 低温度适合事实核查
MAX_TOKENS = 300
```
